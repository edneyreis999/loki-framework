# Model and Effort Guidance for Loki Artifacts

This document is the package reference for choosing model capability class,
reasoning effort and adapter projection metadata for Loki commands, skills,
agents, templates, plans and generated documentation.

Canonical Loki artifacts should describe intent with provider-neutral metadata.
Concrete provider model IDs are an updateable projection, not a permanent rule
to duplicate across every artifact.

## Canonical Fields

Use these fields when an artifact contract needs execution guidance:

```yaml
model_class: frontier_reasoning
effort: high
adapter_projection:
  codex: "Advisory unless projected through config, profile or custom agent."
  claude_code: "May map to model/effort frontmatter where supported."
escalation_signals:
  - durable package policy
  - conflicting evidence
  - high-risk implementation
```

### `model_class`

- `frontier_reasoning`: complex reasoning, durable policy, multi-source
  synthesis, long-horizon workflow orchestration and high-risk decisions.
- `coding`: implementation, code review, debugging, test repair and technical
  proposals that mainly depend on source code.
- `generalist`: balanced analysis, normal documentation, task execution support
  and retrospective synthesis.
- `specialist_generalist_human_like`: reader-facing or editorial documents that
  need natural, polished writing without losing technical accuracy.
- `long_context`: read-only extraction, cataloging and synthesis over many
  local files where coverage matters more than deep invention.
- `fast_low_cost`: simple lookup, routing, triage and low-risk transient notes.

### `effort`

- `low`: quick routing, local lookup, simple status notes, lightweight
  transient documentation and low-risk subagent work.
- `medium`: normal implementation, code review, task execution support,
  retrospective notes and human-readable documentation with known sources.
- `high`: durable framework policy, technical analyses, action plans, complex
  orchestration, conflicting evidence, security or architectural risk.
- `xhigh`: rare escalation for hard multi-phase, multi-source or high-impact
  decisions where latency and cost matter less than correctness.

Do not set `high` everywhere. The value should communicate the real cost and
quality expectation for the artifact.

## Field Mapping

`effort` is the generic Loki concept for reasoning depth, cost and validation
difficulty. Specific artifact contracts can use narrower field names when the
shape of the artifact needs more precision:

- Skills, agents and simple metadata surfaces use `effort`.
- Command contracts use `execution_profile.default_effort` for the normal path
  and `execution_profile.max_effort` for explicit escalation.
- Action plans and technical analyses use `execution_effort` for downstream
  orchestration guidance.
- Task files use `task_effort` for the task itself and `validator_effort` for
  validation work.
- Handoffs can use `handoff_effort` or `recommended_handoffs` to describe
  effort expected from read-only or proposal-only agents.

All of these fields should still use the same `low`, `medium`, `high` and
`xhigh` values defined here.

## Artifact Classification

Durable framework documentation should use larger-capability models and high
effort by default. This includes package policies, guardrails, contracts,
templates, standards, workflow documentation, inventories and documents reused
by many future agents.

Transient documentation produced while executing a task can use low or medium
effort when it only records local state, evidence, build output, resume notes or
human-loop bookkeeping.

Two transient outputs are explicit exceptions and should be produced with high
effort:

- technical analysis generated by `loki:tech-analysis`;
- action plans generated by `loki:generate-action-plan`.

Code implementation should use `model_class: coding` and `effort: medium` by
default. Escalate to `high` for architectural changes, integrations, migrations,
security-sensitive edits, hard root-cause debugging, broad cross-file changes,
runtime behavior that is hard to validate or changes with unclear blast radius.

Human-like documents should use `model_class: specialist_generalist_human_like`
or `generalist`. Use `effort: medium` for normal reader-facing writing and
`effort: high` when the document is durable, normative, source-conflict-heavy or
will guide future agents.

## Recommended Routing

| Work type | Model class | Default effort | Escalate when |
| --- | --- | --- | --- |
| Durable package policy, guardrails and standards | `frontier_reasoning` | `high` | Rule affects many artifacts or relaxes a guardrail. |
| `loki:tech-analysis` output | `frontier_reasoning` | `high` | Conflicting sources, architecture, security or irreversible decision. |
| `loki:generate-action-plan` output | `frontier_reasoning` | `high` | Large plan, complex dependencies or many human gates. |
| Task-local build notes, interaction notes and resume state | `fast_low_cost` or `generalist` | `low` | Note contains decision logic or reusable learning. |
| Code implementation and code review | `coding` | `medium` | High-risk runtime, integration, migration or broad refactor. |
| Read-only context extraction | `long_context` | `low` or `medium` | Many sources, ambiguous ownership or evidence conflicts. |
| Human-facing explanatory docs | `specialist_generalist_human_like` | `medium` | Durable or normative document. |
| Package command, skill, agent and template authoring | `frontier_reasoning` | `high` | Cross-adapter semantics or package policy changes. |

## Adapter Projection

`model_class` and `effort` do not have identical enforcement semantics in every
adapter.

Claude Code can apply model selection and effort through supported skill,
subagent or settings surfaces, subject to configuration precedence and
available model restrictions. Its model configuration supports aliases such as
`fable`, `opus`, `sonnet` and `haiku`, provider-specific pinning and capability
flags for effort support.

Codex should treat `model_class` and `effort` inside Loki `SKILL.md` files as
advisory package metadata unless those values are projected into a Codex
configuration profile, command invocation or custom agent TOML. Codex model
selection is a runtime/configuration concern; package docs must not promise
automatic routing from Markdown frontmatter alone.

When a provider or runtime cannot enforce a field directly, keep the field as
intent metadata and document the adapter-specific projection path.

## Current Provider Projection

This matrix was checked against official OpenAI and Anthropic documentation on
2026-06-26. It is evidence for current projection, not a permanent package
contract. Recheck official provider docs before updating concrete model IDs.

Official sources:

- OpenAI Codex models: `https://developers.openai.com/codex/models`
- OpenAI latest model guide: `https://developers.openai.com/api/docs/guides/latest-model.md`
- Anthropic models overview: `https://docs.anthropic.com/en/docs/about-claude/models/overview`
- Claude Code model configuration: `https://docs.anthropic.com/en/docs/claude-code/model-config`

### OpenAI

| Loki use | Suggested projection | Effort |
| --- | --- | --- |
| Frontier reasoning, durable policy, complex workflows and tool-heavy agents | `gpt-5.5` | `high` or `xhigh` |
| Professional coding, review, plans and agentic workflows | `gpt-5.4` | `medium` or `high` |
| Fast subagents, triage and lighter coding tasks | `gpt-5.4-mini` | `low` or `medium` |
| Near-instant coding iteration where available | `gpt-5.3-codex-spark` | `low` or `medium` |

OpenAI's Codex documentation currently recommends starting most Codex tasks
with `gpt-5.5`, using `gpt-5.4-mini` for faster lower-cost lighter work, and
treating `gpt-5.3-codex-spark` as a research-preview option for rapid coding
iteration.

### Anthropic

| Loki use | Suggested projection | Effort |
| --- | --- | --- |
| Highest available capability and demanding long-horizon reasoning | `Claude Fable 5` / `claude-fable-5` | `high` or `xhigh` |
| Complex reasoning, architecture review and agentic coding | `Claude Opus 4.8` | `high` |
| Daily coding, balanced analysis and technical writing | `Claude Sonnet 4.6` | `medium` or `high` |
| Fast simple tasks, background work and lightweight triage | `Claude Haiku 4.5` | `low` or `medium` |

Anthropic's model overview currently points to Claude Opus 4.8 for complex
tasks and Claude Fable 5 for the highest available capability. Claude Code
supports alias and pinning controls that may affect which concrete model an
environment uses.

## Anti-Patterns

- Duplicating concrete model IDs across every command, skill and agent.
- Treating provider projection as package policy.
- Marking transient bookkeeping as `high` without a specific escalation signal.
- Using a cheap or fast model for durable package policy because the edit looks
  short.
- Claiming Codex or Claude Code enforcement that the active adapter does not
  provide.
- Promoting a task-local observation into durable guidance without source,
  validation and human gate.

## Validation Checklist

Before accepting model/effort metadata in a Loki artifact, check:

- The artifact is classified as durable, transient, implementation,
  human-like, read-only extraction or orchestration.
- `model_class` describes capability, not a provider model ID.
- `effort` matches risk, durability and validation difficulty.
- `adapter_projection` states what can actually be enforced by the target
  adapter.
- Concrete model IDs, when present, appear in an updateable projection table or
  adapter-specific surface rather than as duplicated canonical rules.
